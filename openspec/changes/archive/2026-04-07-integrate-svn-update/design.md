## Context

目前的開發流程中，使用者需手動執行外部的 `update-all.bat` 腳本來同步多個 SVN 子專案。這種方式無法在 App 中直接追蹤進度，且與 App 的整合度低。本設計旨在將此功能內建於 Code7 的核心流程中。

## Goals / Non-Goals

**Goals:**
- 提供「單一更新」與「批次更新」的 UI 入口。
- 實作穩定的 `svn update` 執行引擎，並具備超時處理（120秒）。
- 提供更新過程中的即時進度通知。

**Non-Goals:**
- 不涉及自動 commit 選項。
- 不提供進階的 SVN 參數設定（如 --depth 等）。

## Decisions

### 1. 批次更新的名單管理方式
不再採用固定的硬編碼名單。改為動態從 `Settings.getConfig().projects` 抓取所有使用者已匯入的專案。
- **原因**：提升 App 的彈性與個人化程度。
- **自定義過濾**：在設定面板中為每個專案增加 `isBatchEnabled` 開關（預設為開啟），讓使用者能彈性勾選哪些專案要參與一鍵同步。

### 2. 進度條視覺回饋機制
為了改善原本僅有純文字提示的不足，引入視覺化進度條：
- **單一專案更新**：在 Toast 中顯示 **Indeterminate (循環流動)** 加載條，提供「後台正在作業」的視覺提示。
- **多專案批次更新**：在 Toast 中顯示 **Determinate (按比例增長)** 進度條。寬度根據 `(當前索引 + 1) / 總專案數 * 100%` 計算。
- **實作細節**：重構 `Utils.showToast` 以返回 DOM 引用與更新介面，確保 `SyncController` 能即時控制 UI 變化。

### 3. UI 選項放置位置
在頂部工具列增加「同步 🔄」下拉選單：
- **選項 A**：更新當前專案
- **選項 B**：更新所有子專案 (批量)
- **進度顯示**：更新時在右下角彈出持久性 Toast，包含進度條與專案名稱。

### 4. IPC 進度回饋機制
主進程透過 `webContents.send` 發送 `svn:update-progress` 事件，回傳當前更新的索引、總數與專案名稱。
- **原因**：批次操作需要非同步且持續的狀態更新。
## Risks / Trade-offs

- **[Risk] SVN 衝突中斷** → 更新過程中若發生 conflict，`svn update` 會卡住或回傳錯誤。
  - **Mitigation**: 設定 120 秒超時，並在失敗時回傳 stderr 給 UI 顯示。
- **[Trade-off] 效能** → 同時發啟多個 `svn update` 會佔用網路寬頻與磁碟 IO。
  - **Decision**: 採用**序列式 (Sequential)** 更新，即完成一個再換下一個。
